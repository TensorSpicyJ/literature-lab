# Lean 源码恢复选项

## 目的

这份文档回答一个现在已经无法回避的问题：

- `TopoOrder/` 源文件树不在当前工作树里
- 它也不在当前 repo 的 git 历史里
- 当前只剩聚合入口、文档说明与部分 build 痕迹

在这种情况下，Lean 支线要怎样恢复，哪条路最值得先走？

## 当前结论

当前最合理的恢复优先级是：

1. 外部历史版本 / 备份恢复
2. build 痕迹引导的半手工恢复
3. 基于知识库与路线文档的完全重写

这个排序的原则是：

- 先争取找回原始源码
- 找不回原始源码时，再利用 build 痕迹降低重建成本
- 只有前两条路都不可行时，才进入完全手工重写

## 为什么已经进入“源码恢复”问题

基于当前调查，已经能确认：

- `TopoOrder/` 源文件树不在当前 `14-拓扑序学习库` 工作树里
- git 历史里也没有它们
- `.trace` / `setup.json` 表明这些文件曾在旧路径 `13-拓扑序学习库` 下真实存在并被编译过
- 当前 `13` 路径本身也不再保留这批源码

这说明问题不是普通的 build breakage，而是：

- 一次不完整迁移之后的源码缺席

## Option A：外部历史版本 / 备份恢复

### 描述

从下面这些来源中恢复 `TopoOrder/` 源文件树：

- 百度同步盘历史版本
- 旧机器
- 旧工作树
- 手动备份
- 临时导出目录

### 为什么它是首选

- 能最大概率保留原始源码结构
- 能保留原来的文件切分与命名
- 能避免把已有 toy 与 bridge 重新发明一遍
- 一旦恢复成功，后续只需做校验与清理，而不是重建整个支线

### 适用条件

- 只要任何外部历史来源还可访问，就应该先试这条路

### 风险

- 可能需要人工登录同步盘历史或切换设备
- 可能找回的是较旧版本，而不是最终状态

### 预期成本

- 低到中

### 一旦成功，下一步是什么

- 恢复 `TopoOrder/` 源树
- 跑 `lake build`
- 对照 `TopoOrder.lean` 和 capability map 校验完整性

## Option B：build 痕迹引导的半手工恢复

### 描述

如果原始源码找不回，可以利用现有 build 痕迹作为恢复线索：

- `.trace`
- `setup.json`
- `.olean`
- `TopoOrder.lean`
- `README.md`
- `lean-sandbox-plan.md`

它们不能直接还原源码，但可以帮助恢复：

- 模块名
- 依赖顺序
- 源路径历史
- 一部分声明名与结构层级

### 为什么它排第二

- 它仍然接近原始项目结构
- 比完全重写更容易保持旧设计思路
- 但它已经无法保证逐行恢复原始源码

### 适用条件

- 外部历史恢复失败
- 但 build 痕迹仍然基本完整

### 风险

- 容易“恢复出一个看起来相似、实则不同”的版本
- 某些 theorem 或定义只能靠文档摘要回推，细节会丢

### 预期成本

- 中到高

### 当前已知可用线索

- 模块名列表
- 依赖关系
- 旧路径 `13-拓扑序学习库`
- 某些关键声明名：
  - `TraceAgreementCategory`
  - `closureScalar`
  - `leftTraceBubble`
  - `compositeBubble`

### 当前额外限制

本轮 smoke 显示：

- 现有 `TopoOrder/*.olean` 的导入还会进一步卡在缺失的 `mathlib` `.olean`
- 进一步连 `lake exe cache get` 也会因为 `mathlib/Cache/Main.lean` 缺失而失败
- 即使在独立临时目录里拉起健康 package 层，当前 `TopoOrder/*.olean` 仍报 `incompatible header`

因此当前 Option B 不能简单理解成：

- “直接从现有 `.olean` 里把声明全挖出来”

更准确地说，它目前仍然主要是：

- import 图与模块关系恢复
- 旧路径与依赖顺序恢复

而不是稳定的声明级逆向恢复。

### 一旦走这条路，应该怎样做

1. 先恢复模块骨架与 import 图
2. 再恢复每个文件的一句职责定义
3. 再按 bridge -> toy 顺序逐个重写
4. 每恢复一层，就重新 build 一次

## Option C：基于知识库的完全重写

### 描述

放弃追求恢复旧源码，直接把 Lean 支线当成一个新项目重开。

可用材料包括：

- `archive/lean-sandbox/README.md`
- `lean-sandbox-plan.md`
- `lean-capability-map.md`
- `lean-translation-contract.md`
- `concepts/`
- `roadmap/`

### 为什么它排第三

- 成本最高
- 最容易偏离原项目的真实设计
- 但如果前两条都失败，它仍然是可行的长期路径

### 适用条件

- 外部历史恢复失败
- build 痕迹也不足以支撑有意义的恢复
- 或者决定干脆用更清晰的新结构重启 Lean 支线

### 风险

- 需要重做大量 bridge / toy
- 容易把“文档里的描述”误认为“原来代码里的精确结构”
- 会显著延后 canonical model lane

### 预期成本

- 高

### 这条路的唯一好处

- 可以顺手把旧支线里模糊的边界一次性整理干净

## 当前推荐策略

### 默认策略

当前最合理的默认策略是：

- **先做 Option A**
- A 失败后，再转 **Option B**
- 只有 A/B 都失败，才考虑 **Option C**

### 为什么不直接 C

因为现在已经知道：

- 这批源码曾真实存在
- 还留下了比较丰富的 build 痕迹

这说明直接重写是可以做，但不是最经济的第一选择。

## 决策门槛

### 什么时候放弃 Option A

当下面条件都满足时，可考虑放弃 A：

- 已检查同步盘历史
- 已检查旧机器/旧工作树
- 已检查任何离线备份
- 仍然没有 `TopoOrder/` 源文件树

### 什么时候转入 Option B

当 A 失败，但下面条件成立时：

- `.trace` / `setup.json` 仍然可读
- `TopoOrder.lean` 仍在
- 文档对模块职责的描述足够密
- package 层损坏不会立刻让所有 artifact 勘探失效

### 什么时候被迫进入 Option C

当下面条件同时出现时：

- 找不到外部历史副本
- build 痕迹不足以支持有意义的恢复
- 仍然希望保留 Lean 支线

## 当前建议的最小动作

在不贸然重写的前提下，当前建议的最小动作是：

1. 先把 Option A 当成默认恢复路线。
2. 同时保留 Option B 的恢复框架，不让已有 build 线索浪费。
3. 暂不进入 Option C。
